# Icon Candidate Contract

This directory contains three distinct icon/logo candidate systems for the Year Planner app.
Each candidate provides canonical SVG masters from which preview PNGs and production assets are derived.

## Candidate Folder Contract

Each candidate lives in a folder named `C{N}-{slug}/` and must contain:

| File | Purpose | Dimensions |
|------|---------|-----------|
| `icon.svg` | Square app-icon master | `viewBox="0 0 512 512"` |
| `logo.svg` | Horizontal lockup master | `viewBox="0 0 480 120"` |
| `preview-16.png` | Favicon preview 16Ã—16 | 16Ã—16 px |
| `preview-32.png` | Favicon preview 32Ã—32 | 32Ã—32 px |
| `preview-180.png` | Apple touch icon preview | 180Ã—180 px |
| `preview-192.png` | PWA manifest icon preview | 192Ã—192 px |
| `preview-512.png` | PWA splash / app-store preview | 512Ã—512 px |

Preview PNGs live **directly in the candidate folder** (no `previews/` subdirectory).
They are generated by `bash scripts/export-icon-candidates.sh`.

## Naming Convention

| Segment | Rule |
|---------|------|
| Candidate ID | `C1`, `C2`, `C3` â€” stable, never renumbered |
| Slug | Kebab-case design-system name (e.g. `ink-paper`) |
| Folder | `C{N}-{slug}` (e.g. `C1-ink-paper`) |
| SVG masters | `icon.svg`, `logo.svg` â€” exact names, always lowercase |
| Preview PNGs | `icon-{size}.png` where `{size}` is the pixel dimension of one side |

## SVG Master Requirements

All master SVGs must satisfy:

- Valid `viewBox` attribute (no `width`/`height` attributes at root level â€” let the browser scale)
- Adequate padding so shape edges are not clipped when rasterised at small sizes
- No embedded raster images, no external href references, no `<use>` referencing out-of-file symbols
- `<title>` element immediately inside `<svg>` for accessibility
- Clean path geometry â€” avoid transforms that accumulate precision errors

## Winner Selection Metadata

After candidate review, exactly one winner is recorded in machine-readable JSON:

| File | Purpose |
|------|---------|
| `canonical.json` | The selected winner â€” `selectionStatus: "winner"`, `candidateId`, `folder`, `previews`, `svgSources` |
| `alternatives.json` | The two archived alternatives â€” `selectionStatus: "archived-alternative"` for each non-winner |

`canonical.json` and `alternatives.json` are the authoritative selection artifacts consumed by downstream
export and wiring slices. Do not edit them without updating `data-selection-state` attributes in
`mockups/icon-comparison.html` to match.

## Machine-Checkable Contract

The asset smoke spec (`.tests/smoke/icon-candidates-assets.spec.js`) enforces:

1. Exactly C1, C2, C3 candidate folders exist under `mockups/icon-candidates/`.
2. Each candidate folder contains `icon.svg` and `logo.svg`.
3. Each SVG file is non-empty and contains a `viewBox` attribute.
4. The `icon.svg` viewBox is `0 0 512 512`.
5. The `logo.svg` viewBox is `0 0 480 120`.
6. All 15 preview PNGs exist (`preview-{size}.png` directly in each candidate folder, not in a subdirectory).

The selection smoke spec (`.tests/smoke/icon-candidates-selection.spec.js`) enforces:

1. `canonical.json` exists and contains exactly one winner with a valid candidate ID (C1, C2, or C3).
2. `alternatives.json` exists and contains exactly two `archived-alternative` entries.
3. The winner + two alternatives together cover all three candidate IDs with no overlap.
4. Gallery (`mockups/icon-comparison.html`) `data-selection-state` attributes agree with the JSON metadata.
